## Key Findings

### Critical

1. **Plaintext Passwords + Hardcoded Admin Backdoor** — The authentication system provides no real security. The combination of `admin`/`admin` bypass (`auth.py` lines 102-107) and plaintext password storage and comparison across all CRUD operations (`student_actions.py` line 26, `professor_actions.py` line 27, `auth.py` line 116) means any user who can read the source code (all users, since this repository is public) can authenticate as administrator.

2. **No Server-Side Authorization** — Every data-access function across all five `src/api_actions/` modules is unprotected. The role-based UI hiding in the frontend is the only barrier. A user can modify the browser's network request to perform any operation, including accessing all students, professors, enrollments, evaluations, and performing any CRUD action.

3. **Weak Session Encryption** — The `storage_secret` (`main.py` line 253) is a hardcoded, predictable string. This key encrypts all server-side session data. Compromising this key enables full session forgery.

### High

4. **Brute-Force and Resource Exhaustion** — Login (`auth.py` lines 91-125) and registration (`auth.py` lines 196-231) have no rate limiting, account lockout, or CAPTCHA mechanisms, enabling unlimited automated attacks.

5. **Network Exposure** — Binding to `0.0.0.0` (`main.py` line 250) without TLS (`main.py` line 248-256 configuration) exposes the application to every reachable network interface with all data in plaintext.

### Medium

6. **Data Mass Exposure** — Every `list*` function returns all records regardless of the requesting user's relationship to the data. Client-side filtering is the only protection.

7. **Ubiquitous Stored XSS Risk** — User-supplied text fields (names, comments, descriptions) render directly into Vue/Quasar templates without sanitization, across all entity management pages.
